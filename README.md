## Project Overview

This project implements a real-time distance estimation system using an HC-SR04 ultrasonic sensor and a 2-state discrete Kalman filter running on an STM32 NUCLEO-G474RE microcontroller.

The raw distance readings from the HC-SR04 are inherently noisy. The Kalman filter smooths these readings by maintaining two internal state estimates — **distance** and **velocity** — and blending the noisy measurements with a physics-based prediction model at every sample step.

The filtered output is sent over UART to a Python script on a PC, which plots the raw reading, the filtered distance, and the estimated velocity in real time.

This directly implements the **state estimator** concept from FRA233 Week 10, where the Kalman filter acts as an optimal observer that reconstructs both observable and non-observable states (velocity is never measured directly) from a single noisy sensor.

## Theory — 2-State Kalman Filter

### Why 2 States?

A 1-state filter models the distance as constant between samples:

$$x[k+1] = x[k]$$

It is always surprised when something moves and must "chase" the measurement reactively.

A 2-state filter tracks both **distance** and **velocity**, so it predicts where the object will be next based on how fast it was moving:

$$\text{distance}[k+1] = \text{distance}[k] + \text{velocity}[k] \cdot \Delta t$$
$$\text{velocity}[k+1] = \text{velocity}[k]$$

This makes the estimate proactive rather than reactive.

### System Model

The state vector contains two quantities:

$$\mathbf{x}[k] = \begin{bmatrix} d[k] \\ \dot{d}[k] \end{bmatrix} = \begin{bmatrix} \text{distance (cm)} \\ \text{velocity (cm/s)} \end{bmatrix}$$

The discrete-time system model is:

$$\mathbf{x}[k+1] = \mathbf{A}\,\mathbf{x}[k] + \mathbf{w}[k], \quad \mathbf{w} \sim \mathcal{N}(\mathbf{0},\, \mathbf{Q})$$

$$z[k] = \mathbf{H}\,\mathbf{x}[k] + v[k], \quad v \sim \mathcal{N}(0,\, R)$$

Where the matrices are:

$$\mathbf{A} = \begin{bmatrix} 1 & \Delta t \\ 0 & 1 \end{bmatrix}, \quad \mathbf{H} = \begin{bmatrix} 1 & 0 \end{bmatrix}$$

$$\mathbf{Q} = \begin{bmatrix} q_0 & 0 \\ 0 & q_1 \end{bmatrix}, \quad R \in \mathbb{R}$$

$\mathbf{H} = [1\ \ 0]$ because the sensor measures distance only — velocity is never measured directly. The Kalman filter reconstructs it from the sequence of distance measurements.

### Predict Step (Time Update)

$$\hat{\mathbf{x}}^-[k] = \mathbf{A}\,\hat{\mathbf{x}}[k-1]$$

$$\mathbf{P}^-[k] = \mathbf{A}\,\mathbf{P}[k-1]\,\mathbf{A}^T + \mathbf{Q}$$

The predicted state is pushed forward one step using the physics model. The covariance $\mathbf{P}^-$ grows by $\mathbf{Q}$ because uncertainty accumulates when we have not yet seen a new measurement.

### Update Step (Measurement Update)

$$S[k] = \mathbf{H}\,\mathbf{P}^-[k]\,\mathbf{H}^T + R = p_{00}^- + R \quad \in \mathbb{R}$$

$$\mathbf{K}[k] = \frac{\mathbf{P}^-[k]\,\mathbf{H}^T}{S[k]} = \begin{bmatrix} p_{00}^- / S \\ p_{01}^- / S \end{bmatrix}$$

$$\hat{\mathbf{x}}[k] = \hat{\mathbf{x}}^-[k] + \mathbf{K}[k]\,\bigl(z[k] - \mathbf{H}\,\hat{\mathbf{x}}^-[k]\bigr)$$

$$\mathbf{P}[k] = \bigl(\mathbf{I} - \mathbf{K}[k]\,\mathbf{H}\bigr)\,\mathbf{P}^-[k]$$

### Key Insight — Innovation is Scalar

Even though $\mathbf{P}$ is $2\times2$, the innovation covariance $S$ is a **scalar** because $\mathbf{H}$ is $1\times2$ (only one sensor). This means:

- The Kalman gain inversion is a single division, not a matrix inverse
- No external math library is needed
- The entire filter runs in roughly 20 floating-point operations per sample

## Library Reference

### HC-SR04 Driver (`hcsr04.h` / `hcsr04.c`)

#### Configuration (edit in `hcsr04.h`)

```c
#define HCSR04_TRIG_PIN     GPIO_PIN_9   /* PA9 = Arduino D8 */
#define HCSR04_TRIG_PORT    GPIOA

#define HCSR04_ECHO_PIN     GPIO_PIN_8   /* PA8 = Arduino D7 */
#define HCSR04_ECHO_PORT    GPIOA

#define HCSR04_ECHO_HIGH_TIMEOUT_MS    30U   /* Max wait for echo to go HIGH */
#define HCSR04_ECHO_LOW_TIMEOUT_MS     50U   /* Max wait for echo to go LOW  */
#define HCSR04_TIMEOUT_VALUE           0xFFFFU /* Returned on timeout         */
#define HCSR04_MIN_INTERVAL_MS         60U   /* Min delay between readings   */
```

#### API

```c
void     HCSR04_Init(TIM_HandleTypeDef *htim);
uint16_t HCSR04_Read(void);
```

#### How It Works

**`HCSR04_Init(htim)`** stores the timer handle internally and drives TRIG LOW so the sensor does not fire spuriously on power-up.

**`HCSR04_Read()`** executes the full measurement sequence:

1. Pull TRIG HIGH for 10 μs — this tells the sensor to fire 8 ultrasonic bursts at 40 kHz
2. Pull TRIG LOW
3. Reset the timer counter to zero — this sets the time reference for the echo measurement
4. Wait for ECHO to go HIGH (rising edge = sound has left the sensor), with a 30 ms timeout
5. Capture `value1` — timer count at the rising edge
6. Wait for ECHO to go LOW (falling edge = echo has returned), with a 50 ms timeout
7. Capture `value2` — timer count at the falling edge
8. Calculate distance:

$$\text{distance (cm)} = \frac{(\text{value2} - \text{value1}) \times 17}{1000}$$

This derives from the speed of sound (0.034 cm/μs) divided by 2 for the round trip. Integer multiplication by 17 then division by 1000 avoids floating-point truncation that would zero out small distances.

Both timeout checks use subtraction-based comparison to be safe against `HAL_GetTick()` overflow after ~49 days of uptime:

```c
if ((HAL_GetTick() - timeout_start) >= HCSR04_ECHO_HIGH_TIMEOUT_MS)
```

### Kalman Filter (`kalman.h` / `kalman.c`)

#### Data Structure

```c
typedef struct {
    float dt;    /* Sample period (seconds) — must match HAL_Delay in loop */
    float q0;    /* Process noise variance: distance state                 */
    float q1;    /* Process noise variance: velocity state                 */
    float R;     /* Measurement noise variance (sensor variance)           */
    float x0;    /* State estimate: distance (cm)                          */
    float x1;    /* State estimate: velocity (cm/s)                        */
    float p00;   /* Error covariance P[0][0]                               */
    float p01;   /* Error covariance P[0][1] = P[1][0] (symmetric)        */
    float p11;   /* Error covariance P[1][1]                               */
} Kalman2State_t;
```

The full $2\times2$ covariance matrix $\mathbf{P}$ is stored as three scalars (`p00`, `p01`, `p11`) because $\mathbf{P}$ is always symmetric — storing all four elements would be redundant.

#### API

```c
void  Kalman2_Init(Kalman2State_t *kf, float dt, float q0, float q1,
                   float R, float initial_d, float initial_v);
float Kalman2_Update(Kalman2State_t *kf, float measurement);
float Kalman2_GetVelocity(const Kalman2State_t *kf);
```

#### How `Kalman2_Update()` Works Step by Step

```c
/* PREDICT: state */
float x0_pred = kf->x0 + dt * kf->x1;   /* d + v*dt */
float x1_pred = kf->x1;                  /* v unchanged */

/* PREDICT: covariance  P^- = A*P*A^T + Q  (expanded manually) */
float p00_pred = kf->p00 + dt*(kf->p01 + kf->p01) + (dt*dt)*kf->p11 + kf->q0;
float p01_pred = kf->p01 + dt * kf->p11;
float p11_pred = kf->p11 + kf->q1;

/* UPDATE: innovation  y = z - H*x^- */
float innovation = measurement - x0_pred;

/* UPDATE: innovation covariance  S = H*P^-*H^T + R = p00^- + R */
float S = p00_pred + kf->R;

/* UPDATE: Kalman gain  K = P^-*H^T / S */
float K0 = p00_pred / S;   /* gain for distance state */
float K1 = p01_pred / S;   /* gain for velocity state */

/* UPDATE: corrected state */
kf->x0 = x0_pred + K0 * innovation;
kf->x1 = x1_pred + K1 * innovation;

/* UPDATE: updated covariance  P = (I - K*H)*P^- */
kf->p00 = (1.0f - K0) * p00_pred;
kf->p01 = (1.0f - K0) * p01_pred;
kf->p11 = p11_pred - K1 * p01_pred;
```

The Kalman gain $K$ balances two competing goals:
- **$K$ near 1**: trust the measurement more (happens when $p_{00}^-$ is large relative to $R$)
- **$K$ near 0**: trust the model prediction more (happens when $p_{00}^-$ is small relative to $R$)

Over time, $\mathbf{P}$ converges to a steady-state value and $K$ stabilises. The filter then runs at constant gain, which is the steady-state Kalman filter.

## Main Application (`main.c`)

### Initialisation Sequence

```c
HAL_TIM_Base_Start(&htim1);          /* Start 1 MHz timer for HC-SR04   */
HCSR04_Init(&htim1);                 /* Initialise sensor driver         */

raw_distance_cm = HCSR04_Read();     /* First reading to seed the filter */

Kalman2_Init(&kf_distance,
             KALMAN_DT,              /* 0.06 s = 60 ms loop period       */
             KALMAN_Q0,              /* 0.1 — distance process noise     */
             KALMAN_Q1,              /* 1.0 — velocity process noise     */
             KALMAN_R,               /* 9.0 — sensor measurement noise   */
             (float)raw_distance_cm, /* Seed distance with first reading */
             0.0f);                  /* Seed velocity at 0 cm/s          */
```

Seeding the filter with the first real reading prevents it from starting at 0 cm and spending many iterations pulling the estimate up to the actual distance.

### Main Loop

```c
while (1)
{
    raw_distance_cm = HCSR04_Read();

    if (raw_distance_cm != HCSR04_TIMEOUT_VALUE)
    {
        filtered_distance = Kalman2_Update(&kf_distance,
                                           (float)raw_distance_cm);
        velocity_cms = Kalman2_GetVelocity(&kf_distance);

        printf("RAW:%u  DIST:%.1f  VEL:%.1f\r\n",
               raw_distance_cm,
               filtered_distance,
               velocity_cms);
    }

    HAL_Delay(60);   /* Must match KALMAN_DT = 0.06 s */
}
```

Timeout readings (`0xFFFF`) are skipped entirely — the filter is not updated on bad readings. This prevents the filter from being corrupted by spurious sensor failures.

### Tuning Constants

```c
#define KALMAN_DT    0.06f   /* Sample period — must match HAL_Delay  */
#define KALMAN_Q0    0.1f    /* Distance process noise variance        */
#define KALMAN_Q1    1.0f    /* Velocity process noise variance        */
#define KALMAN_R     9.0f    /* Measurement noise variance             */
```

## Tuning the Kalman Filter

### Step 1 — Measure Sensor Variance (R)

Point the sensor at a stationary wall. Collect 100 raw readings from the terminal and compute the variance. In Python:

```python
import numpy as np
readings = [134, 135, 133, 136, 134, ...]   # paste your readings here
print(f"Variance (R) = {np.var(readings):.2f}")
```

Set the result as `KALMAN_R` in `main.c`.

### Step 2 — Tune Q0 and Q1

| Symptom | Action |
|---|---|
| Filter too slow to follow moving hand | Increase `KALMAN_Q0` |
| Filtered distance still noisy | Decrease `KALMAN_Q0` |
| Velocity estimate is wrong or unstable | Increase `KALMAN_Q1` |
| Velocity estimate is too sluggish | Decrease `KALMAN_Q1` |

### Step 3 — Verify DT Matches the Loop Delay

`KALMAN_DT` must always equal `HAL_Delay(x) / 1000.0`. If you change `HAL_Delay(60)` to `HAL_Delay(100)`, you must also change `KALMAN_DT` to `0.10f`. A mismatch causes the velocity estimate to be scaled incorrectly.

## Results

When working correctly the live graph shows:

- The raw distance signal jumping by several centimetres between samples even when the target is stationary
- The filtered distance staying nearly constant for a stationary target and smoothly tracking a slowly moving one
- The velocity estimate swinging negative when your hand approaches and positive when it moves away, even though velocity is never measured directly — it is estimated purely from the sequence of distance readings by the Kalman filter

This demonstrates the core principle of the state estimator from FRA233 Week 10: reconstructing hidden states (velocity) from observable measurements (distance) using an optimal observer.
